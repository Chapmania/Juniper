18  /// <summary>
19  /// <para>
20  /// This module implements the well-known Heckbert paradigm for color
21  /// quantization. Most of the ideas used here can be traced back to
22  /// Heckbert's seminal paper
23  /// Heckbert, Paul. "Color Image Quantization for Frame Buffer Display",
24  /// Proc. SIGGRAPH '82, Computer Graphics v.16 #3 (July 1982), pp 297-304.
25  /// </para>
26  /// <para>
27  /// In the first pass over the image, we accumulate a histogram showing the
28  /// usage count of each possible color. To keep the histogram to a reasonable
29  /// size, we reduce the precision of the input; typical practice is to retain
30  /// 5 or 6 bits per color, so that 8 or 4 different input values are counted
31  /// in the same histogram cell.
32  /// </para>
33  /// <para>
34  /// Next, the color-selection step begins with a box representing the whole
35  /// color space, and repeatedly splits the "largest" remaining box until we
36  /// have as many boxes as desired colors. Then the mean color in each
37  /// remaining box becomes one of the possible output colors.
38  /// </para>
39  /// <para>
40  /// The second pass over the image maps each input pixel to the closest output
41  /// color (optionally after applying a Floyd-Steinberg dithering correction).
42  /// This mapping is logically trivial, but making it go fast enough requires
43  /// considerable care.
44  /// </para>
45  /// <para>
46  /// Heckbert-style quantizers vary a good deal in their policies for choosing
47  /// the "largest" box and deciding where to cut it. The particular policies
48  /// used here have proved out well in experimental comparisons, but better ones
49  /// may yet be found.
50  /// </para>
51  /// <para>
52  /// In earlier versions of the IJG code, this module quantized in YCbCr color
53  /// space, processing the raw upsampled data without a color conversion step.
54  /// This allowed the color conversion math to be done only once per colormap
55  /// entry, not once per pixel. However, that optimization precluded other
56  /// useful optimizations (such as merging color conversion with upsampling)
57  /// and it also interfered with desired capabilities such as quantizing to an
58  /// externally-supplied colormap. We have therefore abandoned that approach.
59  /// The present code works in the post-conversion color space, typically RGB.
60  /// </para>
61  /// <para>
62  /// To improve the visual quality of the results, we actually work in scaled
63  /// RGB space, giving G distances more weight than R, and R in turn more than
64  /// B. To do everything in integer math, we must use integer scale factors.
65  /// The 2/3/1 scale factors used here correspond loosely to the relative
66  /// weights of the colors in the NTSC grayscale equation.
67  /// If you want to use this code to quantize a non-RGB color space, you'll
68  /// probably need to change these scale factors.
69  /// </para>
70  /// <para>First we have the histogram data structure and routines for creating it.</para>
71  /// <para>
72  /// The number of bits of precision can be adjusted by changing these symbols.
73  /// We recommend keeping 6 bits for G and 5 each for R and B.
74  /// If you have plenty of memory and cycles, 6 bits all around gives marginally
75  /// better results; if you are short of memory, 5 bits all around will save
76  /// some space but degrade the results.
77  /// To maintain a fully accurate histogram, we'd need to allocate a "long"
78  /// (preferably unsigned long) for each cell. In practice this is overkill;
79  /// we can get by with 16 bits per cell. Few of the cell counts will overflow,
80  /// and clamping those that do overflow to the maximum value will give close-
81  /// enough results. This reduces the recommended histogram size from 256Kb
82  /// to 128Kb, which is a useful savings on PC-class machines.
83  /// (In the second pass the histogram space is re-used for pixel mapping data;
84  /// in that capacity, each cell must be able to store zero to the number of
85  /// desired colors. 16 bits/cell is plenty for that too.)
86  /// Since the JPEG code is intended to run in small memory model on 80x86
87  /// machines, we can't just allocate the histogram in one chunk. Instead
88  /// of a true 3-D array, we use a row of pointers to 2-D arrays. Each
89  /// pointer corresponds to a C0 value (typically 2^5 = 32 pointers) and
90  /// each 2-D array has 2^6*2^5 = 2048 or 2^6*2^6 = 4096 entries. Note that
91  /// on 80x86 machines, the pointer row is in near memory but the actual
92  /// arrays are in far memory (same arrangement as we use for image arrays).
93  /// </para>
94  /// <para>Declarations for Floyd-Steinberg dithering.</para>
95  /// <para>
96  /// Errors are accumulated into the array fserrors[], at a resolution of
97  /// 1/16th of a pixel count. The error at a given pixel is propagated
98  /// to its not-yet-processed neighbors using the standard F-S fractions,
99  /// ... (here) 7/16
100  /// 3/16 5/16 1/16
101  /// We work left-to-right on even rows, right-to-left on odd rows.
102  /// </para>
103  /// <para>
104  /// We can get away with a single array (holding one row's worth of errors)
105  /// by using it to store the current row's errors at pixel columns not yet
106  /// processed, but the next row's errors at columns already processed. We
107  /// need only a few extra variables to hold the errors immediately around the
108  /// current column. (If we are lucky, those variables are in registers, but
109  /// even if not, they're probably cheaper to access than array elements are.)
110  /// </para>
111  /// <para>
112  /// The fserrors[] array has (#columns + 2) entries; the extra entry at
113  /// each end saves us from special-casing the first and last pixels.
114  /// Each entry is three values long, one value for each color component.
115  /// </para>
116  /// </summary>

383  /// Prescan some rows of pixels.
384  /// In this module the prescan simply updates the histogram, which has been
385  /// initialized to zeroes by start_pass.
386  /// An output_buf parameter is required by the method signature, but no data
387  /// is actually output (in fact the buffer controller is probably passing a
388  /// null pointer).
389  /// </summary>
390  private void PrescanQuantize(byte[][] input_buf, int in_row, int num_rows)
391  {
392  for (var row = 0; row < num_rows; row++)
393  {
394  var inputIndex = 0;
395  for (var col = m_cinfo.outputWidth; col > 0; col--)
396  {
397  var rowIndex = input_buf[in_row + row][inputIndex] >> C0_SHIFT;
398  var columnIndex = ((input_buf[in_row + row][inputIndex + 1] >> C1_SHIFT) * HIST_C2_ELEMS)
399  + (input_buf[in_row + row][inputIndex + 2] >> C2_SHIFT);
400 
401  /* increment pixel value, check for overflow and undo increment if so. */
402  m_histogram[rowIndex][columnIndex]++;
403  if (m_histogram[rowIndex][columnIndex] <= 0)
404  {
405  m_histogram[rowIndex][columnIndex]--;
406  }
407 
408  inputIndex += 3;
409  }
410  }

413  /// <summary>
414  /// Map some rows of pixels to the output colormapped representation.
415  /// This version performs Floyd-Steinberg dithering
416  /// </summary>
417  private void Pass2FsDither(byte[][] input_buf, int in_row, byte[][] output_buf, int out_row, int num_rows)
418  {
419  var limit = m_cinfo.m_sample_range_limit;
420  var limitOffset = m_cinfo.m_sampleRangeLimitOffset;
421 
422  for (var row = 0; row < num_rows; row++)
423  {
424  var inputPixelIndex = 0;
425  var outputPixelIndex = 0;
426  int dir; /* +1 or -1 depending on direction */
427  int dir3; /* 3*dir, for advancing inputIndex & errorIndex */
428  int errorIndex;
429  if (m_on_odd_row)
430  {
431  /* work right to left in this row */
432  inputPixelIndex += (m_cinfo.outputWidth - 1) * 3; /* so point to rightmost pixel */
433  outputPixelIndex += m_cinfo.outputWidth - 1;
434  dir = -1;
435  dir3 = -3;
436  errorIndex = (m_cinfo.outputWidth + 1) * 3; /* => entry after last column */
437  m_on_odd_row = false; /* flip for next time */
438  }
439  else
440  {
441  /* work left to right in this row */
442  dir = 1;
443  dir3 = 3;
444  errorIndex = 0; /* => entry before first real column */
445  m_on_odd_row = true; /* flip for next time */
446  }
447 
448  /* Preset error values: no error propagated to first pixel from left */
449  /* current error or pixel value */
450  var cur0 = 0;
451  var cur1 = 0;
452  var cur2 = 0;
453  /* and no error propagated to row below yet */
454  /* error for pixel below cur */
455  var belowerr0 = 0;
456  var belowerr1 = 0;
457  var belowerr2 = 0;
458  /* error for below/prev col */
459  var bpreverr0 = 0;
460  var bpreverr1 = 0;
461  var bpreverr2 = 0;
462 
463  for (var col = m_cinfo.outputWidth; col > 0; col--)
464  {
465  /* curN holds the error propagated from the previous pixel on the
466  * current line. Add the error propagated from the previous line
467  * to form the complete error correction term for this pixel, and
468  * round the error term (which is expressed * 16) to an integer.
469  * RIGHT_SHIFT rounds towards minus infinity, so adding 8 is correct
470  * for either sign of the error value.
471  * Note: errorIndex is for *previous* column's array entry.
472  */
473  cur0 = JpegUtils.RIGHT_SHIFT(cur0 + m_fserrors[errorIndex + dir3] + 8, 4);
474  cur1 = JpegUtils.RIGHT_SHIFT(cur1 + m_fserrors[errorIndex + dir3 + 1] + 8, 4);
475  cur2 = JpegUtils.RIGHT_SHIFT(cur2 + m_fserrors[errorIndex + dir3 + 2] + 8, 4);
476 
477  /* Limit the error using transfer function set by init_error_limit.
478  * See comments with init_error_limit for rationale.
479  */
480  cur0 = m_error_limiter[JpegConstants.MAXJSAMPLE + cur0];
481  cur1 = m_error_limiter[JpegConstants.MAXJSAMPLE + cur1];
482  cur2 = m_error_limiter[JpegConstants.MAXJSAMPLE + cur2];
483 
484  /* Form pixel value + error, and range-limit to 0..MAXJSAMPLE.
485  * The maximum error is +- MAXJSAMPLE (or less with error limiting);
486  * this sets the required size of the range_limit array.
487  */
488  cur0 += input_buf[in_row + row][inputPixelIndex];
489  cur1 += input_buf[in_row + row][inputPixelIndex + 1];
490  cur2 += input_buf[in_row + row][inputPixelIndex + 2];
491  cur0 = limit[limitOffset + cur0];
492  cur1 = limit[limitOffset + cur1];
493  cur2 = limit[limitOffset + cur2];
494 
495  /* Index into the cache with adjusted pixel value */
496  var hRow = cur0 >> C0_SHIFT;
497  var hColumn = ((cur1 >> C1_SHIFT) * HIST_C2_ELEMS) + (cur2 >> C2_SHIFT);
498 
499  /* If we have not seen this color before, find nearest colormap */
500  /* entry and update the cache */
501  if (m_histogram[hRow][hColumn] == 0)
502  {
503  FillInverseCMap(cur0 >> C0_SHIFT, cur1 >> C1_SHIFT, cur2 >> C2_SHIFT);
504  }
505 
506  /* Now emit the colormap index for this cell */
507  var pixcode = m_histogram[hRow][hColumn] - 1;
508  output_buf[out_row + row][outputPixelIndex] = (byte)pixcode;
509 
510  /* Compute representation error for this pixel */
511  cur0 -= m_cinfo.m_colormap[0][pixcode];
512  cur1 -= m_cinfo.m_colormap[1][pixcode];
513  cur2 -= m_cinfo.m_colormap[2][pixcode];
514 
515  /* Compute error fractions to be propagated to adjacent pixels.
516  * Add these into the running sums, and simultaneously shift the
517  * next-line error sums left by 1 column.
518  */
519  var bnexterr = cur0; /* Process component 0 */
520  var delta = cur0 * 2;
521  cur0 += delta; /* form error * 3 */
522  m_fserrors[errorIndex] = (short)(bpreverr0 + cur0);
523  cur0 += delta; /* form error * 5 */
524  bpreverr0 = belowerr0 + cur0;
525  belowerr0 = bnexterr;
526  cur0 += delta; /* form error * 7 */
527  bnexterr = cur1; /* Process component 1 */
528  delta = cur1 * 2;
529  cur1 += delta; /* form error * 3 */
530  m_fserrors[errorIndex + 1] = (short)(bpreverr1 + cur1);
531  cur1 += delta; /* form error * 5 */
532  bpreverr1 = belowerr1 + cur1;
533  belowerr1 = bnexterr;
534  cur1 += delta; /* form error * 7 */
535  bnexterr = cur2; /* Process component 2 */
536  delta = cur2 * 2;
537  cur2 += delta; /* form error * 3 */
538  m_fserrors[errorIndex + 2] = (short)(bpreverr2 + cur2);
539  cur2 += delta; /* form error * 5 */
540  bpreverr2 = belowerr2 + cur2;
541  belowerr2 = bnexterr;
542  cur2 += delta; /* form error * 7 */
543 
544  /* At this point curN contains the 7/16 error value to be propagated
545  * to the next pixel on the current line, and all the errors for the
546  * next line have been shifted over. We are therefore ready to move on.
547  */
548  inputPixelIndex += dir3; /* Advance pixel pointers to next column */
549  outputPixelIndex += dir;
550  errorIndex += dir3; /* advance errorIndex to current column */
551  }
552 
553  /* Post-loop cleanup: we must unload the final error values into the
554  * final fserrors[] entry. Note we need not unload belowerrN because
555  * it is for the dummy column before or after the actual array.
556  */
557  m_fserrors[errorIndex] = (short)bpreverr0; /* unload prev errs into array */
558  m_fserrors[errorIndex + 1] = (short)bpreverr1;
559  m_fserrors[errorIndex + 2] = (short)bpreverr2;
560  }

619  /// </summary>
620  private void ComputeColor(Box[] boxlist, int boxIndex, int icolor)
621  {
622  /* Current algorithm: mean weighted by pixels (not colors) */
623  /* Note it is important to get the rounding correct! */
624  long total = 0;

626  long c1total = 0;
627  long c2total = 0;

688  /// <summary>
689  /// Repeatedly select and split the largest box until we have enough boxes
690  /// </summary>
691  private int MedianCut(Box[] boxlist, int numboxes, int desired_colors)
692  {
693  while (numboxes < desired_colors)
694  {
695  /* Select box to split.
696  * Current algorithm: by population for first half, then by volume.
697  */
698  int foundIndex;
699  if (numboxes * 2 <= desired_colors)
700  {
701  foundIndex = FindBiggestColorPop(boxlist, numboxes);
702  }
703  else
704  {
705  foundIndex = FindBiggestVolume(boxlist, numboxes);
706  }
707 
708  if (foundIndex == -1) /* no splittable boxes left! */
709  {
710  break;
711  }
712 
713  /* Copy the color bounds to the new box. */
714  boxlist[numboxes].c0max = boxlist[foundIndex].c0max;
715  boxlist[numboxes].c1max = boxlist[foundIndex].c1max;
716  boxlist[numboxes].c2max = boxlist[foundIndex].c2max;
717  boxlist[numboxes].c0min = boxlist[foundIndex].c0min;
718  boxlist[numboxes].c1min = boxlist[foundIndex].c1min;
719  boxlist[numboxes].c2min = boxlist[foundIndex].c2min;
720 
721  /* Choose which axis to split the box on.
722  * Current algorithm: longest scaled axis.
723  * See notes in update_box about scaling distances.
724  */
725  var c0 = ((boxlist[foundIndex].c0max - boxlist[foundIndex].c0min) << C0_SHIFT) * R_SCALE;
726  var c1 = ((boxlist[foundIndex].c1max - boxlist[foundIndex].c1min) << C1_SHIFT) * G_SCALE;
727  var c2 = ((boxlist[foundIndex].c2max - boxlist[foundIndex].c2min) << C2_SHIFT) * B_SCALE;
728 
729  /* We want to break any ties in favor of green, then red, blue last.
730  * This code does the right thing for R,G,B or B,G,R color orders only.
731  */
732  var cmax = c1;
733  var n = 1;
734 
735  if (c0 > cmax)
736  {
737  cmax = c0;
738  n = 0;
739  }
740 
741  if (c2 > cmax)
742  {
743  n = 2;
744  }
745 
746  /* Choose split point along selected axis, and update box bounds.
747  * Current algorithm: split at halfway point.
748  * (Since the box has been shrunk to minimum volume,
749  * any split will produce two nonempty subboxes.)
750  * Note that lb value is max for lower box, so must be < old max.
751  */
752  int lb;
753  switch (n)
754  {
755  case 0:
756  lb = (boxlist[foundIndex].c0max + boxlist[foundIndex].c0min) / 2;
757  boxlist[foundIndex].c0max = lb;
758  boxlist[numboxes].c0min = lb + 1;
759  break;
760  case 1:
761  lb = (boxlist[foundIndex].c1max + boxlist[foundIndex].c1min) / 2;
762  boxlist[foundIndex].c1max = lb;
763  boxlist[numboxes].c1min = lb + 1;
764  break;
765  case 2:
766  lb = (boxlist[foundIndex].c2max + boxlist[foundIndex].c2min) / 2;
767  boxlist[foundIndex].c2max = lb;
768  boxlist[numboxes].c2min = lb + 1;
769  break;
770  }
771 
772  /* Update stats for boxes */
773  UpdateBox(boxlist, foundIndex);
774  UpdateBox(boxlist, numboxes);
775  numboxes++;
776  }
777 
778  return numboxes;
779  }

1056  /// <summary>
1057  /// Initialize the error-limiting transfer function (lookup table).
1058  /// The raw F-S error computation can potentially compute error values of up to
1059  /// +- MAXJSAMPLE. But we want the maximum correction applied to a pixel to be
1060  /// much less, otherwise obviously wrong pixels will be created. (Typical
1061  /// effects include weird fringes at color-area boundaries, isolated bright
1062  /// pixels in a dark area, etc.) The standard advice for avoiding this problem
1063  /// is to ensure that the "corners" of the color cube are allocated as output
1064  /// colors; then repeated errors in the same direction cannot cause cascading
1065  /// error buildup. However, that only prevents the error from getting
1066  /// completely out of hand; Aaron Giles reports that error limiting improves
1067  /// the results even with corner colors allocated.
1068  /// A simple clamping of the error values to about +- MAXJSAMPLE/8 works pretty
1069  /// well, but the smoother transfer function used below is even better. Thanks
1070  /// to Aaron Giles for this idea.
1071  /// </summary>
1072  private void InitErrorLimit()
1073  {
1074  m_error_limiter = new int[(JpegConstants.MAXJSAMPLE * 2) + 1];
1075  const int tableOffset = JpegConstants.MAXJSAMPLE;
1076 
1077  const int STEPSIZE = ((JpegConstants.MAXJSAMPLE + 1) / 16);
1078 
1079  /* Map errors 1:1 up to +- MAXJSAMPLE/16 */
1080  var output = 0;
1081  var input = 0;
1082  for (; input < STEPSIZE; input++, output++)
1083  {
1084  m_error_limiter[tableOffset + input] = output;
1085  m_error_limiter[tableOffset - input] = -output;
1086  }
1087 
1088  /* Map errors 1:2 up to +- 3*MAXJSAMPLE/16 */
1089  for (; input < STEPSIZE * 3; input++)
1090  {
1091  m_error_limiter[tableOffset + input] = output;
1092  m_error_limiter[tableOffset - input] = -output;
1093  if ((input & 1) != 0)
1094  {
1095  output++;
1096  }
1097  }
1098 
1099  /* Clamp the rest to final output value (which is (MAXJSAMPLE+1)/8) */
1100  for (; input <= JpegConstants.MAXJSAMPLE; input++)
1101  {
1102  m_error_limiter[tableOffset + input] = output;
1103  m_error_limiter[tableOffset - input] = -output;
1104  }
1105  }

1107  /*
1108  * These routines are concerned with the time-critical task of mapping input
1109  * colors to the nearest color in the selected colormap.
1110  *
1111  * We re-use the histogram space as an "inverse color map", essentially a
1112  * cache for the results of nearest-color searches. All colors within a
1113  * histogram cell will be mapped to the same colormap entry, namely the one
1114  * closest to the cell's center. This may not be quite the closest entry to
1115  * the actual input color, but it's almost as good. A zero in the cache
1116  * indicates we haven't found the nearest color for that cell yet; the array
1117  * is cleared to zeroes before starting the mapping pass. When we find the
1118  * nearest color for a cell, its colormap index plus one is recorded in the
1119  * cache for future use. The pass2 scanning routines call fill_inverse_cmap
1120  * when they need to use an unfilled entry in the cache.
1121  *
1122  * Our method of efficiently finding nearest colors is based on the "locally
1123  * sorted search" idea described by Heckbert and on the incremental distance
1124  * calculation described by Spencer W. Thomas in chapter III.1 of Graphics
1125  * Gems II (James Arvo, ed. Academic Press, 1991). Thomas points out that
1126  * the distances from a given colormap entry to each cell of the histogram can
1127  * be computed quickly using an incremental method: the differences between
1128  * distances to adjacent cells themselves differ by a constant. This allows a
1129  * fairly fast implementation of the "brute force" approach of computing the
1130  * distance from every colormap entry to every histogram cell. Unfortunately,
1131  * it needs a work array to hold the best-distance-so-far for each histogram
1132  * cell (because the inner loop has to be over cells, not colormap entries).
1133  * The work array elements have to be ints, so the work array would need
1134  * 256Kb at our recommended precision. This is not feasible in DOS machines.
1135  *
1136  * To get around these problems, we apply Thomas' method to compute the
1137  * nearest colors for only the cells within a small subbox of the histogram.
1138  * The work array need be only as big as the subbox, so the memory usage
1139  * problem is solved. Furthermore, we need not fill subboxes that are never
1140  * referenced in pass2; many images use only part of the color gamut, so a
1141  * fair amount of work is saved. An additional advantage of this
1142  * approach is that we can apply Heckbert's locality criterion to quickly
1143  * eliminate colormap entries that are far away from the subbox; typically
1144  * three-fourths of the colormap entries are rejected by Heckbert's criterion,
1145  * and we need not compute their distances to individual cells in the subbox.
1146  * The speed of this approach is heavily influenced by the subbox size: too
1147  * small means too much overhead, too big loses because Heckbert's criterion
1148  * can't eliminate as many colormap entries. Empirically the best subbox
1149  * size seems to be about 1/512th of the histogram (1/8th in each direction).
1150  *
1151  * Thomas' article also describes a refined method which is asymptotically
1152  * faster than the brute-force method, but it is also far more complex and
1153  * cannot efficiently be applied to small subboxes. It is therefore not
1154  * useful for programs intended to be portable to DOS machines. On machines
1155  * with plenty of memory, filling the whole histogram in one shot with Thomas'
1156  * refined method might be faster than the present code --- but then again,
1157  * it might not be any faster, and it's certainly more complicated.
1158  */
1159 
1160  /*
1161  * The next three routines implement inverse colormap filling. They could
1162  * all be folded into one big routine, but splitting them up this way saves
1163  * some stack space (the mindist[] and bestdist[] arrays need not coexist)
1164  * and may allow some compilers to produce better code by registerizing more
1165  * inner-loop variables.
1166  */
1167 
1168  /// <summary>
1169  /// Locate the colormap entries close enough to an update box to be candidates
1170  /// for the nearest entry to some cell(s) in the update box. The update box
1171  /// is specified by the center coordinates of its first cell. The number of
1172  /// candidate colormap entries is returned, and their colormap indexes are
1173  /// placed in colorlist[].
1174  /// This routine uses Heckbert's "locally sorted search" criterion to select
1175  /// the colors that need further consideration.
1176  /// </summary>
1177  private int FindNearbyColors(int minc0, int minc1, int minc2, byte[] colorlist)
1178  {
1179  /* Compute true coordinates of update box's upper corner and center.
1180  * Actually we compute the coordinates of the center of the upper-corner
1181  * histogram cell, which are the upper bounds of the volume we care about.
1182  * Note that since ">>" rounds down, the "center" values may be closer to
1183  * min than to max; hence comparisons to them must be "<=", not "<".
1184  */
1185  var maxc0 = minc0 + ((1 << BOX_C0_SHIFT) - (1 << C0_SHIFT));
1186  var centerc0 = (minc0 + maxc0) >> 1;
1187 
1188  var maxc1 = minc1 + ((1 << BOX_C1_SHIFT) - (1 << C1_SHIFT));
1189  var centerc1 = (minc1 + maxc1) >> 1;
1190 
1191  var maxc2 = minc2 + ((1 << BOX_C2_SHIFT) - (1 << C2_SHIFT));
1192  var centerc2 = (minc2 + maxc2) >> 1;
1193
